# UI Layout - Patterns & Rules
> Load when touching sidebar, layouts (workspace vs project), ProjectPicker, CommandPalette, release dialog, or version badge.
> Last validated: 2026-04-30

## Layout Tree

```
AppLayout              SidebarProvider wrapper (protected shell) вҖ” outermost
 в”ңв”Җ WorkspaceLayout    Sidebar (workspace nav) + breadcrumb header + <Outlet>
 в”Ӯ   в””в”Җ Projects page
 в”ңв”Җ NewProject         full-width, reuses AppLayout only
 в””в”Җ ProjectLayout      Sidebar (project nav, with Settings collapsible) + breadcrumb header + <Outlet>
      в””в”Җ Project pages (Overview, Deployments, Integrations/*, Settings/*)
```

All three layouts share `AppSidebar` (`components/layout/AppSidebar.tsx`) with a `context: 'workspace' | 'project'` prop. Project context also needs `projectId`.

## `AppSidebar` Responsibilities

| Slot | Content |
|------|---------|
| Header (`SidebarHeader`) | Workspace context вҶ’ Deployik logo + "Dashboard" subtitle. Project context вҶ’ `<ProjectPicker />` (command-based switcher) |
| Content group 1 | Nav items: workspace (Projects, Account Tokens) OR project (Overview, Deployments, Integrations with Analytics/Email/Multi Locale, Settings with Build/Domains/Environments/Protection) |
| Content group 2 | Quick actions: "All Projects" (project context), "New Project" |
| Footer (`SidebarFooter`) | `<VersionRow />` (SHA + build badge) above the user avatar menu вҶ’ workspace radio group + Log Out |

The Integrations and Settings items use `<Collapsible defaultOpen>` with `SidebarMenuSub` children. Each subitem can carry `data-testid` for Playwright selectors; use the `testId` field in `NavSubItem` instead of hand-editing the rendered `Link`.

## Command Patterns

| Component | Trigger | Behavior |
|-----------|---------|----------|
| `ProjectPicker` (sidebar header, project context) | Click current project pill | Command list of all projects; Enter navigates to `/projects/{id}` |
| `CommandPalette` (global Cmd/Ctrl+K) | Keyboard shortcut | Actions + workspaces + project search; single entry point to anywhere |

Both use `cmdk` via the shadcn `command` primitive.

## Release Dialog (post-redesign, commit 2c624ae)

**Shape:** `ReleasePanelContent` (`components/projects/release-panel.tsx`) is a **pure content component**, not a standalone panel. It is rendered inside:
- `ProjectOverview.tsx` вҖ” inside a `<Dialog>` triggered by the "Release to production" action.
- `ProjectDeployments.tsx` вҖ” same dialog pattern for triggering a production deployment from the list view.

Props:
```ts
{
  project: Project;
  domains: Domain[] | undefined;
  createTag: boolean;
  onCreateTagChange: (value: boolean) => void;
  releaseTagName: string;
  onReleaseTagChange: (value: string) => void;
}
```

The parent page owns the mutation (`api.triggerDeployment(projectId, { environment: 'production', create_tag, tag_name })`) and handles close-on-success.

**Default tag name** is built by `buildReleaseTagName(new Date())` from `deployment-helpers.ts` вҶ’ `release-YYYYMMDD-HHmm`.

## shadcn Sidebar Primitives (`components/ui/sidebar.tsx`)

Bundled from shadcn; used throughout:
- `SidebarProvider`, `Sidebar`, `SidebarInset`, `SidebarTrigger`, `SidebarRail`
- `SidebarHeader`, `SidebarContent`, `SidebarFooter`
- `SidebarGroup`, `SidebarGroupLabel`
- `SidebarMenu`, `SidebarMenuItem`, `SidebarMenuButton`
- `SidebarMenuSub`, `SidebarMenuSubItem`, `SidebarMenuSubButton`

Variant/collapsible: `<Sidebar variant="sidebar" collapsible="icon">` вҖ” collapses to icon rail on narrow screens.

## Do / Don't

| Do | Don't | Why |
|----|-------|-----|
| Use `SidebarMenuButton asChild + <Link>` for nav items | Nest `<button>` inside `<Link>` | Accessibility + styling breaks otherwise |
| Drive `isActive` from `useRouterState().location.pathname` via a `matchPath(pathname)` closure | Compare location.pathname inline in JSX | Sub-route patterns like `/settings/*` are cleaner as predicates |
| Keep project context in the sidebar via `context + projectId` props | Thread routes through global state | Component stays pure, easy to reuse |
| Reuse `components/ui/breadcrumb.tsx` in `ProjectLayout` header | Build a custom breadcrumb | Consistent spacing/typography |
| For a new dialog, pair with existing `Dialog` primitive + a Content component in `components/projects/` | Inline 200-line dialogs inside pages | Matches the `ReleasePanelContent` pattern |

## Gotchas

- **`TopBar.tsx` still exists** (from pre-redesign) but is NOT wired into any layout. Do not import it; if you do, it will double-render nav. Slated for removal.
- **`Projects` page lives inside `WorkspaceLayout`** BUT `NewProject` uses `AppLayout` directly (no sidebar-nav items exist for it). The project-nav sidebar is only for `/projects/$id/*`.
- **`ProjectPicker` currentProjectName** falls back to `"..."` while the project query is loading вҖ” that's intentional; don't show a spinner (looks worse).
- **Dark theme only** вҖ” `index.css` / `styles.css` set zinc palette. Light theme is not implemented; any component assuming `dark:` prefix toggling will misbehave.
- **SidebarProvider cookies** вҖ” shadcn's sidebar persists collapsed state in a cookie by default. If that conflicts with SSR/auth cookies, check cookie names.

# Analytics - Patterns & Rules
> Load when touching Umami audience stats, Loki runtime logs, or the project analytics/integration UI.
> Last validated: 2026-04-16

## Two Data Sources

| Source | For | Backend client | Shape |
|--------|-----|----------------|-------|
| Umami | Audience (pageviews, visitors, referrers) | `internal/analytics/umami.go` | Per-project website in Umami, segmented preview vs production by host/domain filter |
| Loki | Runtime logs (requests, bandwidth, latency, API paths) | `internal/analytics/loki.go` | Reads nginx JSON access logs: `/var/log/nginx/deployik-<project_id>-<name>-<env>.json` |

`analytics.Service` in `service.go` orchestrates both and returns a combined payload for `GET /api/projects/{id}/analytics`.

## Provisioning

One Umami website per project (tracked in `project_analytics` table, FK 1:1 with `projects`). Provisioning is **best-effort**:

- On project create/update: attempt to create the Umami website + patch `project_analytics` row.
- On analytics read: if `project_analytics.audience_status != 'receiving_data'`, lazy-retry provisioning.
- Existing projects without a row get one auto-created on first analytics access (no migration backfill needed).

`AnalyticsAudienceStatus` values (see `models.go`): `provisioning`, `ready_to_install`, `waiting_for_data`, `receiving_data`, `stale`, `unavailable`, `error`.

## Install-with-AI Pattern

`components/projects/project-integration.tsx` surfaces two tabs under the Integration route:

1. **AI install** — a generated prompt that embeds project framework, package manager, root directory, and the linked Umami website ID. User copies it into any coding assistant which adds the snippet to their repo. Uses `ui/code-panel.tsx`.
2. **Manual snippet** — the raw `<script>` tag with the Umami script URL and website ID for hand-installation.

**Important:** Deployik does NOT inject scripts via proxy. This was an explicit design choice — brittle, breaks on Next.js `output: 'standalone'`, blocks CSPs. Install-with-AI replaces that approach.

## Tracker vs API URL Split

Umami install supports a separate tracker script URL from the events POST URL:
- `AnalyticsUmamiScriptURL` — served from Lovinka CDN (fast, cacheable)
- `AnalyticsUmamiPublicURL` — Umami API that receives events

This lets the tracker load from our edge while still posting events back to the Umami backend.

## Reusable UI Primitives

| Component | Purpose | Used by |
|-----------|---------|---------|
| `components/analytics/stat-card.tsx` | KPI card with CardAction + CardFooter | Overview stats, Audience summary |
| `components/analytics/metric-chart.tsx` | shadcn chart-card wrapper around Recharts | Pageviews timeseries, Bandwidth timeseries |
| `components/ui/code-panel.tsx` | Fixed-height scrollable code/prompt card with copy action | Install-with-AI prompts, manual snippets |
| `components/projects/project-analytics-meta.ts` | `AUDIENCE_STATUS_META` map: badge style + description per status | Both analytics and integration pages |

**Rule:** For any new dashboard chart, reuse `metric-chart.tsx`. For any new KPI, reuse `stat-card.tsx`. Do not introduce one-off chart wrappers.

## Endpoints

```
GET  /api/projects/{id}/analytics?environment=&range=&timezone=
POST /api/projects/{id}/analytics/verify?environment=&range=&timezone=  (forces refresh / retries provisioning)
```

Ranges + timezone normalization live in `options.go`.

## Do / Don't

| Do | Don't | Why |
|----|-------|-----|
| Reuse `stat-card` and `metric-chart` for any new dashboard | Build one-off chart wrappers inline | DRY — matches global CLAUDE.md reusability rule |
| Let provisioning stay best-effort (catch + record status) | Fail the project create on Umami errors | Analytics is optional; project creation must succeed |
| Read `AUDIENCE_STATUS_META[status]` for badge/description | Hardcode per-status strings in components | Status copy must stay consistent across pages |
| Tell the user to re-verify via the Verify endpoint on `ready_to_install` | Auto-retry every page load | Umami rate limits; verify is a user action |

## Gotchas

- **Loki URL can be empty** in dev — `NewLokiClient` handles the zero client and returns empty slices rather than errors.
- **Umami credentials live in config** (`AnalyticsUmamiURL/Username/Password`) — not per-user. The bot login uses them to provision websites.
- **`project_analytics.verified_at`** is set only after a successful stats query, not just provisioning. It drives the `receiving_data` vs `waiting_for_data` badge.
- **Environment filter in Umami** is by **hostname** — the service derives the right hostname list from `domains` table for the requested environment, then filters Umami results in-memory. No separate Umami website per environment.

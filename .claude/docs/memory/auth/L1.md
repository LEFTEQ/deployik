# Auth - Patterns & Rules
> Load when touching login, sessions, tokens, or protected-route wiring.
> Last validated: 2026-04-16

## Session Model

- **Access token**: short-lived JWT (HS256, 1h), signed with `JWT_SECRET`, claims = `{user_id, github_id, username, role}`. Stored ONLY in the `deployik_access_token` HttpOnly cookie.
- **Refresh token**: opaque random string (32 bytes base64url), hashed with SHA-256 before persisting to `refresh_tokens`. Cookie name: `deployik_refresh_token`. TTL: 7 days. **Rotated on every refresh.**
- **Site-auth cookie**: `deployik_site_auth` — HMAC-SHA256 signed `projectID:env:issuedAt`, 24h TTL. Scoped to password-protected deployments only.

## Conventions

| Pattern | Rule |
|---------|------|
| Cookie names | `auth.AccessCookieName` / `auth.RefreshCookieName` (see `internal/auth/session.go`). Never hardcode strings. |
| Cookie flags | `HttpOnly: true`, `SameSite=Lax`, `Secure` controlled by `cfg.FrontendCookieSecure()` (derived from `FRONTEND_URL`). |
| OAuth state | Random nonce set in short-lived `deployik_oauth_state` cookie; verified in `GithubCallback` before exchanging the code. |
| Claims access | `auth.GetClaims(r.Context())` — returns `*Claims` or nil. Only populated after `middleware.Authenticate`. |
| Hashing | `auth.HashRefreshToken(raw)` → SHA-256 hex. Stored columns hold **only the hash**, never the raw token. |

## Do / Don't

| Do | Don't | Why |
|----|-------|-----|
| Use `api.ts` client (frontend) which sets `credentials: 'include'` | Store JWT in localStorage / Zustand persist | Golden rule — tokens stay in HttpOnly cookies |
| Call `/api/auth/me` via `hydrateAuthState()` in route `beforeLoad` | Assume auth from localStorage | localStorage is cleared on explicit logout only; cookies are the source of truth |
| Rotate refresh token on every `/auth/refresh` | Return the same token | Replay protection — old hash is revoked |
| Record login/refresh/logout in `audit_logs` | Silent auth | Security audit visibility |
| Add new protected endpoints inside `r.Group(func(r chi.Router){ r.Use(middleware.Authenticate(...)) ... })` | Add auth check inside the handler | Consistency + easier to audit |

## Authorization (see `internal/authz/access.go`)

Two rules:
1. `claims.Role == "admin"` → bypass, `database.GetProject(id)` directly.
2. Regular user → `database.GetProjectForUser(id, userID)` which matches the project creator OR any member of the project's organization.

**Admin bootstrap:** controlled by `ADMIN_GITHUB_USERS` env (comma-separated usernames). If that env is set, ONLY those usernames are promoted to admin. If it is empty, the **very first signed-up user** is auto-promoted as a fallback. Subsequent users stay `role='user'`.

## DEV_MODE Affordances

- `POST /api/auth/dev-login {username?}` — creates/returns a dev user, sets cookies, no GitHub round-trip. Used by Playwright E2E.
- `handlers.ProjectHandler.DevMode == true` triggers mock repo/branch lists for `/api/github/repos` and `/api/github/branches`.
- `cfg.FrontendCookieSecure()` returns false when `FRONTEND_URL` is `http://` so dev cookies actually set.

## Endpoints (from `router.go`)

```
GET  /api/auth/github            Redirect to GitHub OAuth (scope: repo,read:user,admin:repo_hook)
GET  /api/auth/github/callback   Verify state → exchange code → upsert user → issue cookies
POST /api/auth/refresh           Rotate refresh cookie, mint new access
POST /api/auth/logout            Revoke refresh row, clear both cookies
GET  /api/auth/me                Return current user
POST /api/auth/dev-login         DEV_MODE only
```

## Gotchas

- **`admin:repo_hook` scope is required** for the auto-build feature to create webhooks. Users who logged in before that scope was added must re-authorize.
- **Cookie domain is not set** — relies on default (exact host). If `FRONTEND_URL` and API live on different subdomains, auth breaks. Solved today by same-origin deploy.
- **OAuth state cookie is not `Secure` in dev** but IS in production. If you debug callbacks over plain HTTP, check `FrontendCookieSecure()`.
- **`cfg.AllowedGithubUsers`** (allowlist) is checked in `GithubCallback` — unlisted users get a 403 redirect. Empty list = allow all (default dev behavior).

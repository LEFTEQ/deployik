# Workspaces (Organizations) - Patterns & Rules
> Load when touching multi-user / multi-workspace behavior, personal orgs, memberships, or the workspace switcher.
> Last validated: 2026-04-16

## Model

| Table | Purpose |
|-------|---------|
| `organizations` | Named workspace. `is_personal=1` + `personal_owner_user_id=<user_id>` for per-user auto-orgs. |
| `organization_memberships` | `(org_id, user_id, role in ['owner','member'])`. Grants project visibility. |
| `projects.organization_id` | Nullable FK. If NULL, project is in the creator's personal org (backfilled by migration 006). |

## Invariants

- **Every user has exactly one personal organization** — created lazily by `EnsurePersonalOrganization(userID)` in `queries_organizations.go`. Called from `UpsertUser` after OAuth callback and from `GetUserByID` on first access after migration.
- **`personal_owner_user_id` is unique** (one personal org per user). FK to `users.id`.
- **Shared orgs have `is_personal=0`** and get visibility via `organization_memberships` rows (role `owner` or `member`).
- **Project creator always retains access** regardless of org membership changes (enforced in `GetProjectForUser`).

## Authorization Join (`GetProjectForUser`)

A regular user sees a project iff:
```sql
projects.user_id = :user_id                            -- they created it
  OR projects.organization_id IN (
       SELECT organization_id FROM organization_memberships WHERE user_id = :user_id
     )
```

Admins bypass via `authz.LoadProject` (`claims.Role == "admin"` → `GetProject(id)`).

## Endpoints

```
GET  /api/organizations                       List orgs the current user can access (personal + shared)
GET  /api/projects?organization_id=<org>      List projects filtered to one workspace
POST /api/projects {organization_id?, ...}    Defaults to personal org if omitted; must be a member otherwise
```

Organization CRUD (create shared orgs, invite members) is NOT exposed in the current API — only personal orgs auto-exist, shared orgs come from manual DB inserts for now. Flag for future work.

## Frontend Wiring

| Layer | Concern |
|-------|---------|
| `store/organization.ts` | Zustand + persist; stores `selectedOrganizationId` only |
| `hooks/use-organizations.ts` | React Query over `/api/organizations` + bridges the selected id from Zustand |
| `components/layout/AppSidebar.tsx` footer | Workspace radio group — writes through `setSelectedOrganizationId` |
| `pages/Projects.tsx` | Filters list by `selectedOrganizationId` (passes it to `api.listProjects`) |
| `pages/NewProject.tsx` | Sends `organization_id` on create |

On logout: `useOrganizationStore.getState().clearSelection()` runs BEFORE `clearAuth()` to avoid a flash of stale data.

## Do / Don't

| Do | Don't | Why |
|----|-------|-----|
| Always go through `GetProjectForUser(id, userID)` for regular users | Direct `GetProject(id)` | Bypasses the membership join |
| Pass `organization_id` on create, let the handler default to personal if empty | Require the UI to know the personal org id | Server owns the default |
| Expose `organization_name` on `Project` responses when listing | Force the client to re-query orgs | Handy for the Projects list chip |
| Check membership before accepting `organization_id` on `POST /api/projects` | Trust the client | Users could assign to any org by id |

## Gotchas

- **Migration 006 backfills `projects.organization_id`** with the creator's personal org id. New orgs post-migration simply insert a new row.
- **`EnsurePersonalOrganization`** uses the user's `username` as a slug seed (lowercased, de-duped). If two users have similar usernames the slug picks up a numeric suffix.
- **The workspace switcher is a radio group**, not a link list — switching doesn't navigate, it just re-filters the current page via the persisted selection. Expect a refetch.
- **"Organizations" as a concept** here is NOT tied to GitHub orgs. A Deployik organization is purely a Deployik workspace; GitHub repos are still owned at the user level (via the project creator's token).
- **`GET /api/organizations` returns `membership_role`** per org — `"owner"` for your personal org and for shared orgs you own, `"member"` otherwise. UI uses this to gate destructive actions.

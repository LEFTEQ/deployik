# Variables (Env + Secrets) - Patterns & Rules
> Load when touching env vars, secrets, the VariableStore UI, or how values reach a deployed container.
> Last validated: 2026-04-16

## One Table, Two Stores

`env_variables` holds BOTH env vars and secrets. They are distinguished by `kind` column:

| | Env (`kind='env'`) | Secret (`kind='secret'`) |
|---|---|---|
| Scopes | `shared`, `preview`, `production` | `shared`, `preview`, `production` |
| Build-time injection | `NEXT_PUBLIC_*` baked as `ENV` lines in Dockerfile builder | NEVER build-time |
| Runtime injection | Passed as `docker run -e` | Passed as `docker run -e` |
| `NEXT_PUBLIC_*` keys allowed? | Yes | **No** — API rejects with 400 |
| API response | Value masked (`****last4`) | Value masked |
| Same key in both stores for same project | Not allowed — handler rejects on conflict | Same |

Unique index: `UNIQUE(project_id, environment, key)` — scope-per-env-per-project. A key may exist in the same scope across multiple projects.

## Shared → Scoped Merge

Pipeline resolution order (enforced in SQL inside `queries_envvars.go`):

1. Load all rows where `environment='shared'`.
2. Load all rows where `environment=<requested>` (preview or production).
3. Scoped rows **override** shared by `key`.

The resulting list is returned by `ListResolvedEnvVars` / `ListResolvedSecrets` — both are used by the build pipeline.

## Handler (`internal/api/handlers/envvars.go` — `VariableHandler`)

**One handler struct serves both stores** — the `Kind` field (`VariableKindEnv` | `VariableKindSecret`) is set at wire-up in `router.go`. Routes:

```
GET    /api/projects/{id}/env?environment=           list env vars (masked)
PUT    /api/projects/{id}/env                        BULK set (replaces all for scope)
POST   /api/projects/{id}/env                        single UPSERT (additive)
DELETE /api/projects/{id}/env/{key}?environment=     remove one key

/api/projects/{id}/secrets/*                         same shape, Kind=secret
```

**Bulk vs upsert:**
- `BulkSet` replaces the entire `(project_id, environment, kind)` slice — use when the UI has a full snapshot (.env import).
- `Upsert` adds/updates a single key — used by the Vercel-style row-at-a-time UI.

Conflict detection in both paths: cross-store check — if the same key exists in the other store for the same project, the request is rejected.

## Frontend (`components/projects/variable-store.tsx`)

Single component for both env and secret stores. Props include `kind`, `title`, and the list/mutation callbacks. Features:

- Inline add / edit / delete rows
- Scope badge (shared / preview / production) via `VARIABLE_SCOPE_META` in `deployment-helpers.ts`
- `.env` paste import — parses KEY=VALUE lines, detects `NEXT_PUBLIC_*`, rejects for secrets store
- Reveal is server-driven: only masked values come back from the API; user cannot see existing plaintext

Used from:
- `ProjectSettingsEnv.tsx` — renders two `<VariableStore>` blocks (env + secrets)

## Do / Don't

| Do | Don't | Why |
|----|-------|-----|
| Use `BulkSet` for `.env` imports (single PUT, single set-of-writes) | Loop `Upsert` calls from the client | Race + N round-trips |
| Encrypt on write, decrypt on read, mask in API | Store plaintext "for debugging" | Golden rule — nothing plaintext at rest |
| Reject `NEXT_PUBLIC_*` in the secret store at the API layer | "Clean up" in the pipeline | Early rejection is testable + obvious |
| Reject a key that exists in the other store | Silently overwrite | Two semantics for the same key would confuse users |
| Fetch `kind` from the path, not the body | Accept `kind` in JSON | URL is the source of truth; avoids spoofing |

## Gotchas

- **Mask format is `****last4`** — 4 asterisks + last 4 chars of the ciphertext post-decrypt. For short values (<4 chars) you get all asterisks.
- **`environment` on bulk set is required** — no implicit "update all scopes". Missing → 400.
- **Deleting a key is scoped.** `DELETE /env/{key}?environment=preview` doesn't touch the `shared` or `production` row with the same key.
- **The `ProjectVariable.Value` field serializes as the masked form via `json:"value"`**, but the DB column holds ciphertext. The masking happens in `List*` query functions, not in JSON encoding.
- **`NEXT_PUBLIC_*` in the env store only affects Next.js builds** (baked into the client bundle). For Vite/Astro you are responsible for referencing `import.meta.env.*` or equivalent at build time.

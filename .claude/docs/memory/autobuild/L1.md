# Auto-Build - Patterns & Rules
> Load when touching GitHub webhooks, push-triggered deployments, auto-on-create behaviour, or admin-scope error UX.
> Last validated: 2026-04-30

## Model

One `auto_build_configs` row per project (unique on project_id). Fields:

```
enabled             bool
production_branch   string   default "main"   (exact match)
preview_branches    string   default "*"      (comma-separated, or "*" = all)
auto_production_enabled bool default false     (production follows pushes to production_branch only when opted in)
webhook_id          *int64   GitHub webhook ID (nil before first enable)
webhook_secret      string   AES-256-GCM ciphertext
```

Idempotency is tracked in `webhook_events` via unique `(github_delivery_id, project_id, environment)`. One GitHub delivery can legitimately create both preview and production deployments when production auto-deploy is opted in.

## Setup Flow (`PUT /api/projects/{id}/auto-build`)

1. Load authorized project; load project **owner's** decrypted GitHub token (NOT the current user — see gotchas).
2. If no existing webhook_id → call `provisionWebhook(...)` (package-level helper in `autobuild.go`).
3. If existing config → preserve `auto_production_enabled` when the JSON field is omitted; explicit `false` is meaningful.
4. If existing config → only flip the GitHub webhook active state on enable/disable toggles (`UpdateWebhookActive`).
5. Record audit entry `autobuild.update`.

## Auto-setup on Project Creation (`handlers/projects.go`)

`ProjectHandler.Create` fires best-effort goroutine-free calls **after** the project row is committed:

1. **`setupAutoBuildBestEffort`** — decrypts creator's token, calls `provisionWebhook` with defaults: `production_branch=project.Branch`, `preview_branches="*"`, `auto_production_enabled=req.auto_production_enabled`. On failure (scope, network, decrypt) it logs and returns nil; project is created regardless.
2. **`dispatchInitialDeployBestEffort(..., "preview")`** — inserts a preview `deployments` row (`trigger_source=api`) and calls `Pipeline.Dispatch`. Nil-Pipeline is a no-op (defensive).
3. **Optional production initial deploy** — only runs when auto-build setup returned a persisted config with `Enabled && AutoProductionEnabled && ProductionBranch == project.Branch`.

New-project defaults: `auto_build_enabled` defaults true, `auto_production_enabled` defaults false. `auto_build_enabled=false` still creates the project and initial preview deploy, but skips webhook/config setup and therefore never queues production.

Both emit audit events with `metadata={"source":"project_create"}`.

`provisionWebhook` is the **shared helper** used by both `Put` (user-driven enable) and the creation-time path. Its signature:
```go
func provisionWebhook(ctx, db, encryptor, project, token, webhookURL, productionBranch, previewBranches string, autoProductionEnabled bool) (*db.AutoBuildConfig, error)
```

### Error Mapping (critical UX fix from commits 37c739b + 3d3f0d4)

GitHub responses from `CreateWebhook` are mapped to specific HTTP status + payload shape:

| GitHub status | Deployik returns | `error` code | `message` |
|---------------|------------------|--------------|-----------|
| 404 | 403 | `no_admin_access` | "You need admin access to {owner}/{repo} on GitHub to create webhooks. Ask the repo owner to add you as an admin collaborator." |
| 403 | 403 | `insufficient_scope` | "Re-authorize with GitHub to enable auto-build." |
| other | 502 | (no code) | "failed to create webhook: {err}" |

Frontend distinguishes these via the `error` code and renders different CTAs ("Ask for admin" vs "Re-authorize").

## Webhook Handler (`handlers/webhooks.go` + `POST /api/webhooks/github`)

1. Read body, parse push event (`ref`, `after`, `head_commit.message`, `pusher.name`), and load active configs for the repo.
2. Validate `X-Hub-Signature-256` HMAC against each candidate project's decrypted webhook secret until one matches.
3. Do not write `webhook_events` until after signature validation. Delivery idempotency is claimed per target environment below.
4. Match branch against the project's config:
   - `preview_branches == "*"` OR in comma-separated list → target environment `preview`.
   - `auto_production_enabled && branch == production_branch` → target environment `production`.
   - Both can match; this intentionally blurs preview and production for simple one-branch sites.
   - Neither → record event as `ignored`, return 200.
5. Claim a `webhook_events` row with `status="received"` before creating each deployment. Duplicate redeliveries for the same `(delivery, project, environment)` return 200 without creating another deployment.
6. Create `deployment` with `trigger_source="webhook"`, `triggered_by_username=pusher`, then call `Pipeline.Dispatch`.
7. Update the claimed `webhook_events` row to `processed` (deployment id attached) or `failed`.

## Do / Don't

| Do | Don't | Why |
|----|-------|-----|
| Decrypt **owner**'s token to manage the webhook | Use current user's token | Webhook survives user churn; owner is stable |
| Include `active` in the GH webhook when toggling enabled | Delete + recreate | Keeps the same webhook_id → stable delivery history |
| Return 200 on ignored events | Return 4xx | GitHub retries 4xx/5xx — we don't want retries for "this branch doesn't match" |
| Check HMAC before writing `webhook_events` | Claim delivery rows before signature validation | Invalid signatures are ignored without recording events; delivery IDs are only persisted for validated webhooks |
| Gate production on `auto_production_enabled` | Infer production from `production_branch` alone | Production auto-deploy is explicit opt-in; manual releases still use `production_branch` |

## Gotchas

- **OAuth scope `admin:repo_hook` is mandatory** for `CreateWebhook`. Users who authorized before this scope was added get the 403 `insufficient_scope` path. Login must be re-done.
- **404 from GitHub ≠ repo missing** — for webhook endpoints it means "you are not an admin on this repo". Deployik maps this to its own 403 with `no_admin_access`.
- **Deleting auto-build also deletes the GitHub webhook** (`DELETE /api/projects/{id}/auto-build`). Failures during delete are logged but don't block the DB delete (the webhook may already be gone from GitHub).
- **`WEBHOOK_URL` env** defaults to `{FRONTEND_URL}/api/webhooks/github`. In production make sure it is publicly reachable from GitHub (i.e. not `localhost`).
- **One project matches at most one `auto_build_configs` row** (UNIQUE project_id). If you need per-branch rules beyond one production branch + preview pattern list, that is a future enhancement.
- **Production opt-in does not create tags** — webhook-triggered production deploys use the pushed commit directly. Manual release dialog remains the tag-creation path.
- **Wildcard `*` for `preview_branches`** is matched before the comma-list check; `main,dev` will NOT match `feature/foo`.
- **Auto-setup does not block project creation** — failures in `setupAutoBuildBestEffort` are swallowed. The user can retry by visiting Settings → Auto-build.
- **`trigger_source="api"` for creation-time deploy** — same enum value used by the REST `POST /deployments` endpoint. Distinct from `"webhook"` and `"manual"`.
- **`provisionWebhook` is shared** — when modifying the webhook-creation logic (secret length, event types, active state), update `autobuild.go:provisionWebhook`; changes are automatically inherited by both user-driven setup AND creation-time setup.

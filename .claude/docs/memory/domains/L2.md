## `ProvisionConfig`

```go
type ProvisionConfig struct {
    ProjectID         string
    ProjectName       string
    Domain            string   // canonical
    RedirectDomain    string   // www variant, empty if none
    SSLDomains        []string // for certbot SAN: all variants
    Environment       string   // "preview" | "production"
    ContainerName     string   // Docker name the proxy forwards to
    PasswordProtected bool
}
```

`Manager.ProvisionDomain(cfg, stream bool, logger *ProvisionLogger)` is idempotent:
- If cert exists and is fresh в†’ certbot `--keep-until-expiring` no-ops.
- nginx config is rewritten each call; `-t` catches template regressions before reload.

      try_files /auth.html =404;

## Password Protection Flow

1. **Enable** (`PUT /api/projects/{id}/protection {env, enabled: true}`):
   - Generate `base64.RawURLEncoding(rand_bytes(12))` в‰€ 16 chars.
   - Encrypt with `crypto.Encryptor.Encrypt`.
   - `DB.SetProjectPassword(projectID, env, ciphertext)` в†’ persists to `projects.preview_password` or `production_password`.
   - For each domain in env: re-render nginx config with `PasswordProtected: true` в†’ test в†’ reload.
   - Return plaintext password **once** in the response body.

2. **User visits a protected URL**:
   - nginx `location /` fires `auth_request /_deployik/auth-check`.
   - That location proxies to `/api/site-auth/check`, passing `X-Deployik-Project` + `X-Deployik-Environment`.
   - `ProtectionHandler.Check` verifies the `deployik_site_auth` cookie HMAC against `JWT_SECRET` and the stored password hash.
   - Missing/invalid cookie в†’ 401 в†’ nginx `error_page 401 = @auth_required` serves `/var/www/html/auth.html`.

3. **User submits the form**:
   - POST to `/_deployik/verify` (nginx passes through to `/api/site-auth/verify`).
   - `ProtectionHandler.Verify` decrypts stored password, compares, issues `deployik_site_auth` cookie (HS256 of `projectID:env:issuedAt`, 24h TTL, `HttpOnly`, `Secure`, `SameSite=Lax`).
   - Response 302 to the `redirect` form field (the URL they were going to).

4. **Regenerate** (`POST /api/projects/{id}/protection/regenerate {env}`):
   - New password, new ciphertext, same flow as enable.
   - Old `deployik_site_auth` cookies STILL WORK until they naturally expire (we don't maintain a version salt). Frontend wording calls this out.

5. **Disable** (`PUT /api/projects/{id}/protection {env, enabled: false}`):
   - `DB.ClearProjectPassword(projectID, env)`.
   - Re-render nginx configs without the auth_request blocks.

## Variant Resolution (`ResolveVariantPlan`)

| Input | Environment | Canonical | Redirect | AllDomains |
|-------|-------------|-----------|----------|-----------|
| `app.example.com` | production | `app.example.com` | вЂ” | `[app.example.com]` |
| `example.com` | production | `example.com` | `www.example.com` | `[example.com, www.example.com]` |
| `www.example.com` | production | `example.com` | `www.example.com` | `[example.com, www.example.com]` |
| `*` | preview | input as-is | вЂ” | `[input]` |

Production apex is always primary so SEO canonicals and cookies land on `example.com`. Preview environments skip variant logic entirely.

## Certbot Invocation

```
docker run --rm \
  -v $PROXY_CERTS_DIR:/etc/letsencrypt \
  -v $PROXY_HTML_DIR:/var/www/html \
  certbot/certbot certonly --webroot \
  --webroot-path /var/www/html \
  --email $SSL_EMAIL \
  --agree-tos --no-eff-email \
  --non-interactive \
  --keep-until-expiring \
  -d <domain1> [-d <domain2> ...]
```

`--keep-until-expiring` makes the call idempotent вЂ” certbot skips when the cert has >30 days remaining.

## Edge Cases

- **DNS propagation delay:** `VerifyDomainDNS` does a single `net.LookupHost`. If DNS still returns old IP, user must retry. We don't poll.
- **`AAAA` records are ignored.** Only A records are checked against `VPSHost`. Pure IPv6 setups are unsupported today.
- **Cert request for a new subdomain when the apex cert exists:** certbot issues a separate cert for the subdomain. Both configs get their own `ssl_certificate` path.
- **Wildcard certs** are not supported вЂ” we only use webroot (HTTP-01) challenges. Wildcards need DNS-01 which requires DNS API integration (future work).
- **Nginx test failure (`nginx -t` exits non-zero):** we log the stderr, return an error, DO NOT reload. The previous config stays live.
- **Auth page served from one shared path** (`$PROXY_HTML_DIR/auth.html`) вЂ” same Czech page for all protected sites. One file, all projects.

# Domains & SSL - Patterns & Rules
> Load when touching custom domains, nginx configs, SSL provisioning, or per-environment password protection.
> Last validated: 2026-04-16

## Lifecycle

```
POST /api/projects/{id}/domains      →  row inserted (dns_verified=false, ssl_status=pending)
POST /api/projects/{id}/domains/{did}/verify
      → domain.Manager.ProvisionDomain(config, stream=true, ProvisionLogger):
           1. Verify DNS A-record contains VPS IP (every variant: apex, www)
           2. Run certbot in Docker (bind-mount certs/html)
           3. Render nginx template → write `/opt/nginx-proxy/conf.d/{project}-{env}.conf`
           4. `nginx -t` then `nginx -s reload`
           5. Update domains.ssl_status = 'active', dns_verified = true
```

Streaming verification logs go through the WS hub on `GET /ws/domains/{did}/logs` (see `ws/domain_logs.go`).

## Auto vs Custom Domains

| | Auto | Custom |
|---|------|--------|
| Created when | project create | `POST /api/projects/{id}/domains` |
| `is_auto` | 1 | 0 |
| Delete allowed | **No** (API rejects) | Yes |
| Format | `{project_name}.preview.example.com` (preview) + one for production later | user-supplied |
| DNS verify required | No (we control the zone) | Yes |

## Domain Variants (`internal/domain/variants.go`)

Production custom domains canonicalize: apex is **primary**, `www.<apex>` becomes an optional alias that 301-redirects to apex.

```go
plan := domain.ResolveVariantPlan(domainName, environment)
// plan.CanonicalDomain, plan.RedirectDomain, plan.AllDomains()
```

Use `plan.AllDomains()` for the certbot SAN list and DNS verification; use `plan.CanonicalDomain` as the nginx `server_name` primary.

## Nginx Template (`internal/domain/nginx.go`)

| Block | When | Purpose |
|-------|------|---------|
| HTTP→HTTPS redirect | always | 443 only serves app |
| `location /.well-known/acme-challenge/` | always | certbot renewal |
| `location /_deployik/auth-check` (internal) | password protected | proxy to `/api/site-auth/check` with project/env headers |
| `location /_deployik/verify` | password protected | POST target for the login form |
| `location = @auth_required` | password protected | serves `auth.html` on 401 |
| Main `location /` | always | proxies to container; adds `auth_request /_deployik/auth-check` when protected |
| JSON access log | always | `/var/log/nginx/deployik-<project_id>-<name>-<env>.json` — shipped to Loki |

**Password protection flow:** nginx's `auth_request` returns 401 → served static Czech auth page (`domain.WriteAuthPage` writes it at startup to `$PROXY_HTML_DIR/auth.html`). User POSTs password to `/_deployik/verify` → Deployik issues HMAC-signed `deployik_site_auth` cookie (24h, HS256 over `projectID:env:issuedAt`) → subsequent requests pass `auth_request`.

## Do / Don't

| Do | Don't | Why |
|----|-------|-----|
| Reload nginx with `-t` THEN `-s reload` | Reload without testing first | A bad template drops ALL sites, not just yours |
| Use `--keep-until-expiring` with certbot | Re-issue on every deploy | Rate-limited by Let's Encrypt + wastes time |
| Call `Manager.ProvisionDomain` from the pipeline AND the domain-verify handler | Duplicate the certbot logic | Single entry point, shared `ProvisionLogger` |
| Regenerate nginx when protection is toggled (`handlers/protection.go`) | Hand-edit configs on VPS | Direct VPS edits cause drift (see global CLAUDE.md) |
| Run `ReconcileActiveConfigs` on startup | Rely on old configs surviving redeploys | Nginx confd is a volume but template changes need rewrite |

## Reconcile on Startup

`domain.ReconcileActiveConfigs(manager, targets)` runs from `main.go` boot — rewrites configs for every domain with `ssl_status='active'` so template changes take effect next nginx reload. Targets come from `db.ListActiveDomainProvisionTargets()`.

## Password Protection Endpoints

```
GET  /api/projects/{id}/protection                          →  {preview_enabled, production_enabled}
PUT  /api/projects/{id}/protection {environment, enabled}   →  returns plaintext password when enabling
POST /api/projects/{id}/protection/regenerate {environment} →  returns new plaintext password
POST /api/site-auth/verify                                   →  called by nginx (form POST from auth.html)
GET  /api/site-auth/check                                    →  called by nginx auth_request (cookie only)
```

Enabling protection: generate 16-char base64url password → encrypt → store in `projects.preview_password` or `production_password` → rebuild nginx config with `auth_request` blocks → `nginx -s reload`. Plaintext is returned ONCE in the API response.

## Gotchas

- **`auth.html` path** is shared (`$PROXY_HTML_DIR/auth.html`) — one file serves all protected projects. Localized to Czech; regenerate via `domain.WriteAuthPage` if you touch the template.
- **`deployik_site_auth` cookie is scoped to the project+environment** via the HMAC payload, not via cookie `Path`. A single cookie unlocks all paths on that origin; that is intentional since each origin maps to exactly one project-env pair.
- **`VPSHost` for DNS verification** comes from config (`VPS_HOST=203.0.113.10`). Changing VPS IP requires updating both DNS verifier and platform handler (`/api/platform` returns it for DNS-setup UI).
- **`nginx -s reload`** is issued INSIDE the running nginx container (`docker exec nginx-proxy`) — do not try to exec nginx on the host.
- **SSL cert paths** follow certbot's default (`/etc/letsencrypt/live/<domain>/`), bind-mounted to `$PROXY_CERTS_DIR`.

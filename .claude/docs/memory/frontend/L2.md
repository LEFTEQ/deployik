```

  └── App (app/app.tsx)
        QueryClientProvider (staleTime 30s, retry 1)
        RouterProvider
          └── rootRoute
               TooltipProvider + <Outlet /> + <Toaster />
               ├─ /login                     Login page
               ├─ /auth/callback             AuthCallback (code+state exchange)
               └─ protectedRoute (beforeLoad → hydrateAuthState → redirect if !auth)
                   └── AppLayout (SidebarProvider)
                       ├─ workspaceLayoutRoute (WorkspaceLayout — sidebar context="workspace")
                       │    └── /                → Projects
                       ├─ newProjectRoute → NewProject      (under AppLayout, no project sidebar)
                       └─ projectLayoutRoute (ProjectLayout — sidebar context="project" with projectId)
                            ├─ /projects/$id              → ProjectOverview
                            ├─ /projects/$id/deployments  → ProjectDeployments
                            ├─ /projects/$id/deployments/$did → DeploymentDetail
                            ├─ /projects/$id/integrations/analytics → ProjectAnalytics
                            ├─ /projects/$id/integrations/email     → ProjectEmail
                            ├─ /projects/$id/settings     → ProjectSettings (Build)
                            ├─ /projects/$id/settings/domains    → ProjectSettingsDomains
                            ├─ /projects/$id/settings/env        → ProjectSettingsEnv

```

## Refresh Retry

`ApiClient.request()` catches 401 → fires `/api/auth/refresh` once → retries original request. Second 401 → `useAuthStore.getState().clearAuth()` + `window.location.href = "/login"`. No refresh queue — only one refresh flight at a time via a module-level promise (mirror of `authBootstrapPromise` in `app.tsx`).

## Auth Hydration

`hydrateAuthState()` is the single source of truth for "am I logged in?":

```ts
async function hydrateAuthState() {
  const state = useAuthStore.getState();
  if (state.status !== "unknown") return;   // already bootstrapped
  if (!authBootstrapPromise) {
    authBootstrapPromise = api.getMe()
      .then(u => useAuthStore.getState().setAuthenticated(u))
      .catch(() => useAuthStore.getState().clearAuth())
      .finally(() => { authBootstrapPromise = null; });
  }
  await authBootstrapPromise;
}
```

Used by BOTH `loginRoute.beforeLoad` (redirect authed → `/`) and `protectedRoute.beforeLoad` (redirect unauthed → `/login`). Never called inside a component's render.

## WebSocket Hook Pattern

Both `useBuildLogs` and `useDomainVerification`:

1. Open WS to `/ws/.../logs?token=` (token read from cookie OR session helper).
2. Append lines to a local ring buffer (max N ~= 2000).
3. Expose `{ lines, status, error }`.
4. Close on unmount + on manual close.
5. Auto-reconnect with backoff if the deployment/verification is still in progress.

Origin is verified server-side — in dev, `http://localhost:5173` must be in `ALLOWED_ORIGINS`.

## Edge Cases

- **Race between page-load and auth bootstrap:** `hydrateAuthState` dedupes with a module-level promise; repeated `beforeLoad` calls await the same promise.
- **Token refresh during a WS connection:** WS uses the cookie set at connection time. If the cookie refreshes mid-stream, the next reconnect picks up the new cookie (no explicit refresh flow inside WS hooks).
- **Sidebar collapsed state persisted in a cookie** by shadcn — survives reloads, but SSR-unaware (we don't SSR).
- **Workspace switch while viewing a project-scoped page:** changing the selected workspace doesn't navigate. Project pages load by `$id` and will 404 if the user moves to a workspace they can no longer see. Handle by catching 404 and redirecting to `/`.
- **Deleted project still linked in logs:** soft delete keeps the row; Project pages load fine. Guard elsewhere (list queries filter `status != 'deleted'`).
